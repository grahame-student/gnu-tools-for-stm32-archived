# Binutils Autotools Modernization

## Overview

This document describes the autotools modernization process for the binutils component of the GNU Tools for STM32 toolchain. The binutils stage builds the GNU assembler, linker, and other binary utilities needed for ARM embedded development.

## Component Information

- **Component**: GNU Binutils
- **Source Directory**: `src/binutils/`
- **Build Stage**: Stage 2 (binutils-gcc-first)
- **Build Script**: `build-binutils-gcc-first.sh`
- **Dockerfile Stage**: `binutils-gcc-first`

## Autotools Usage

### Build System Architecture

Binutils uses a hybrid autotools approach:

1. **Top Level**: Uses `autogen` to generate `Makefile.in` from `Makefile.def`
   - Does NOT use automake at top level
   - Uses autoconf 2.69 (required for reproducibility)
   - Makefile.def defines the build structure
   - Makefile.tpl provides the template

2. **Subdirectories**: Many subdirectories use automake
   - `bfd/`, `gas/`, `ld/`, `binutils/`, `gold/`, etc. have `Makefile.am`
   - These generate `Makefile.in` via automake during build
   - Some subdirectories (gdb, libiberty, libdecnumber) use hand-crafted Makefile.in

### Autotools Versions

- **autoconf**: 2.69 (REQUIRED - exact version for reproducibility)
- **automake**: 1.16+ (Ubuntu 24.04 provides 1.16.5)
- **autogen**: Any version (for Makefile.def processing)
- **libtool**: Any recent version

**Why autoconf 2.69?**
Binutils, GCC, GDB, and Newlib require exactly autoconf 2.69 for reproducible builds. Using autoconf 2.71+ will generate different configure scripts and cause build inconsistencies. The build system automatically detects and uses `autoconf2.69` when available.

## Build Process

### Regeneration During Build

The `regenerate_autotools()` function in `build-common.sh` handles binutils regeneration:

1. **Check for existing files** (lines 274-278):
   - If `configure` and `Makefile.in` exist, skip regeneration
   - Prevents unnecessary rebuilds

2. **Generate Makefile.in from Makefile.def** (lines 310-318):
   ```bash
   if [ -f "Makefile.def" ] && which autogen > /dev/null 2>&1; then
       autogen Makefile.def
   fi
   ```

3. **Use autoconf2.69** (lines 293-306):
   ```bash
   if [ "$lib_name" = "binutils" ]; then
       export AUTOCONF=autoconf2.69
       export AUTOHEADER=autoheader2.69
       export AUTOM4TE=autom4te2.69
       export AUTORECONF=autoreconf2.69
       export AUTOUPDATE=autoupdate2.69
       autoreconf_cmd="autoreconf2.69"
   fi
   ```

4. **Install auxiliary files** (lines 343-354):
   - Copies install-sh, missing, config.guess, config.sub, mkinstalldirs, etc.
   - From automake's libdir to binutils root
   - These files are needed for the build system but not tracked in git

5. **Special handling for binutils libtool files** (lines 312-329):
   - Runs `libtoolize` to generate ltmain.sh
   - Manually copies libtool m4 files (libtool.m4, ltoptions.m4, etc.) to top-level
   - Required because binutils configure.ac uses `m4_include([libtool.m4])` without path
   - These files must exist before aclocal runs

6. **Run autoreconf** (lines 343-347):
   ```bash
   autoreconf2.69 -i -f
   ```

7. **Regenerate subdirectories** (lines 351-367):
   - Finds all subdirectories with configure.ac
   - Runs autoreconf on each
   - Generates Makefile.in from Makefile.am in subdirectories

### Build Script Flow

From `build-binutils-gcc-first.sh`:

```bash
# 1. Regenerate autotools files
regenerate_autotools $SRCDIR/$BINUTILS

# 2. Configure binutils
$SRCDIR/$BINUTILS/configure \
    --build=$BUILD \
    --host=$HOST_NATIVE \
    --target=$TARGET \
    --prefix=$INSTALLDIR_NATIVE \
    --enable-plugins \
    --disable-nls \
    --enable-deterministic-archives

# 3. Build and install
make -j$JOBS
make install
```
