# Newlib Autotools Modernization

## Overview

This document describes the autotools modernization process for the newlib component of the GNU Tools for STM32 toolchain. Newlib is the C standard library for embedded systems, providing libc, libm, and runtime libraries for ARM targets.

## Date
2025-11-12

## Component Information

- **Component**: Newlib (C Standard Library for Embedded Systems)
- **Source Directory**: `src/newlib/`
- **Build Stage**: Stage 3 (newlib)
- **Build Script**: `build-newlib.sh`
- **Dockerfile Stage**: `newlib`

## Autotools Usage

### Build System Architecture

Newlib uses a hybrid autotools approach:

1. **Top Level**: Uses `autogen` to generate `Makefile.in` from `Makefile.def`
   - Does NOT use automake at top level
   - Uses autoconf 2.69 (required for reproducibility)
   - Makefile.def defines the build structure
   - Makefile.tpl provides the template

2. **Subdirectories**: Different subdirectories use different approaches
   - `etc/`, `libgloss/`, `newlib/` subdirectories use autoconf/automake
   - These generate `configure`, `Makefile.in`, and `aclocal.m4` during build
   - Some libgloss target-specific directories use hand-crafted Makefile.in files

### Autotools Versions

- **autoconf**: 2.69 (REQUIRED - exact version for reproducibility)
- **automake**: 1.16+ (Ubuntu 24.04 provides 1.16.5)
- **autogen**: Any version (for Makefile.def processing)
- **libtool**: Any recent version

**Why autoconf 2.69?**
Binutils, GCC, GDB, and Newlib require exactly autoconf 2.69 for reproducible builds. Using autoconf 2.71+ will generate different configure scripts and cause build inconsistencies. The build system automatically detects and uses `autoconf2.69` when available.

### Rationale

**Why track hand-crafted Makefile.in files?**
- These are not generated by automake; they are manually written build templates
- They contain custom build logic specific to each target architecture
- They are integral source files, not derived artifacts
- Tracking them ensures they are versioned and not accidentally regenerated

**Why not track autogenerated configure/Makefile.in?**
- `configure` is generated from `configure.ac` by autoconf
- Top-level and subdirectory `Makefile.in` is generated from `Makefile.def` by autogen
- `aclocal.m4` is generated from m4 macros by aclocal
- `config.h.in` is generated from `configure.ac` by autoheader
- These can be reliably regenerated during build using `regenerate_autotools()`

## Build Process

### How Newlib is Built

The build process (in `build-newlib.sh`) follows these steps:

1. **Regenerate autotools files**:
   ```bash
   regenerate_autotools $SRCDIR/$NEWLIB
   ```
   This function (from `build-common.sh`):
   - Runs `autogen Makefile.def` to generate top-level `Makefile.in`
   - Runs `autoreconf2.69 -i -f` to generate `configure` and other files
   - Regenerates subdirectory configure scripts and Makefile.in files
   - Uses autoconf 2.69 specifically for newlib (reproducibility requirement)

2. **Configure newlib**:
   ```bash
   ./configure --target=arm-none-eabi --prefix=... \
       --enable-newlib-io-long-long \
       --enable-newlib-io-long-double \
       --enable-newlib-register-fini \
       --disable-newlib-supplied-syscalls \
       --disable-nls
   ```

3. **Build and install**:
   ```bash
   make -j$JOBS
   make install
   ```

### Integration with Docker Build

The Dockerfile's newlib stage:
```dockerfile
FROM binutils-gcc-first AS newlib
WORKDIR /root/build/gnu-tools-for-stm32
RUN chmod +x build-newlib.sh && \
    ./build-newlib.sh
```

This stage:
- Inherits the GCC first pass compiler from the previous stage
- Runs `build-newlib.sh` which regenerates autotools files automatically
- Builds newlib with size optimization flags (-Os)
- Installs libc.a, libm.a, libg.a, and startup files

### 2. No Changes to configure.ac Required

**Analysis**:
- Newlib's `configure.ac` is maintained by upstream newlib project
- The build system already uses autoconf 2.69 via `regenerate_autotools()`
- No deprecated macros that affect build (minor warnings only)
- Making minimal changes to maintain upstream compatibility

**Decision**: Do not modify newlib's configure.ac to avoid diverging from upstream

### 3. No Changes to Makefile.def Required

**Analysis**:
- `Makefile.def` defines the multi-component build structure
- Already processed correctly by `autogen` during build
- No modernization needed for this file format

**Decision**: Leave Makefile.def as-is

## Comparison with Other Components

### Similarities with Binutils/GCC/GDB

Like binutils, gcc, and gdb, newlib:
- Uses `autogen` to generate top-level `Makefile.in` from `Makefile.def`
- Requires autoconf 2.69 exactly for reproducibility
- Has hand-crafted `Makefile.in` files in some subdirectories
- Regenerates autotools files during build via `regenerate_autotools()`

### Differences from Bootstrap Libraries (GMP, MPFR, MPC, ISL, Expat)

Unlike bootstrap libraries:
- Bootstrap libraries use automake at top level (have Makefile.am)
- Bootstrap libraries use modern autoconf 2.71 (not 2.69)
- Bootstrap libraries track zero autogenerated files
- Newlib tracks 15 hand-crafted Makefile.in files

## Best Practices

### When to Regenerate Autotools Files

Regenerate autotools files when:
- `configure.ac` is modified
- `Makefile.def` is modified
- `Makefile.am` (in subdirectories) is modified
- Switching autotools versions
- After updating newlib from upstream

### How to Regenerate Manually

```bash
cd src/newlib

# Generate top-level Makefile.in from Makefile.def
autogen Makefile.def

# Generate configure and other files
autoconf2.69

# Regenerate subdirectories (optional, done by regenerate_autotools)
cd libgloss && autoconf2.69 && cd ..
cd newlib && autoconf2.69 && cd ..
```

**Important**: Always use `autoconf2.69` (not `autoconf`) for newlib!

### Verifying Hand-Crafted Files

To verify a Makefile.in is hand-crafted:
```bash
# Check for absence of Makefile.am
ls -la src/newlib/libgloss/cris/Makefile.am  # Should not exist

# Check for absence of automake signature
grep "generated by automake" src/newlib/libgloss/cris/Makefile.in  # Should not match
```
