# Automake/Autoconf Linting and Drift Checks

## Overview

This document describes the linting and drift-checking approach for autotools-based build components in the GNU Tools for STM32 toolchain. These checks ensure that autoconf/automake configurations are analyzed for warnings and deprecated usage, and that autogenerated files tracked in version control are in sync with their source files.

## Purpose

The linting and drift checks serve the following goals:

1. **Early Detection**: Identify configuration issues, deprecated syntax, and potential build problems before expensive build steps
2. **Code Quality**: Track autoconf and automake best practices and highlight areas for improvement
3. **Consistency**: Verify that autogenerated files (configure, Makefile.in, etc.) are properly synchronized with their sources
4. **Security**: Identify potential issues in shell scripts through static analysis
5. **Continuous Improvement**: Provide visibility into technical debt and modernization opportunities

## Operational Modes

The linting system supports two modes:

### Informational Mode (Default)

In informational mode, linting runs and reports all issues found, but does **not** fail the build. This mode is useful for:
- Tracking existing technical debt without blocking development
- Identifying areas for gradual improvement
- Learning about autotools best practices
- Monitoring the impact of changes on code quality

**Usage**: `lint_autotools <directory>` or `lint_autotools <directory> --informational`

### Strict Mode

In strict mode, any linting warnings or errors will cause the linting process to fail. This mode is useful for:
- Enforcing quality standards on new code
- Preventing regression in already-clean modules
- CI gates for critical components

**Usage**: `lint_autotools <directory> --strict`

## Linting Tools

### 1. autoconf --warnings=all

**Purpose**: Lint autoconf configuration files (configure.ac, configure.in)

**Command**: `autoconf --warnings=all -o /dev/null`

**Checks for**:
- Deprecated macros and syntax
- Obsolete constructs
- Cross-compilation issues
- Portability problems
- Syntax errors in configure.ac

**Common Warnings**:
- `AC_PROG_CC` used after `AC_INIT` - Order of macro calls
- Deprecated `AC_TRY_*` macros - Should use `AC_*_IFELSE` variants
- Obsolete macros that have modern replacements
- Missing or incorrect macro arguments

**Remediation**:
- Update deprecated macros to their modern equivalents (see autoconf documentation)
- Reorder macro calls to follow autoconf conventions
- Add missing macro arguments
- For binutils/gcc/gdb/newlib: Use autoconf 2.69 specifically (required for reproducible builds)

### 2. automake --warnings=all

**Purpose**: Lint automake Makefile templates (Makefile.am)

**Command**: `automake --warnings=all --dry-run`

**Checks for**:
- Deprecated automake syntax
- Missing required files (NEWS, README, AUTHORS, ChangeLog)
- Portability issues in Makefiles
- Incorrect variable usage
- Missing dependencies

**Common Warnings**:
- `INCLUDES` is deprecated - Use `AM_CPPFLAGS` instead
- Missing required files for GNU projects
- Non-portable Make syntax
- Incorrect automake conditional usage
- Missing `SUBDIRS` or incorrect ordering

**Remediation**:
- Replace `INCLUDES` with `AM_CPPFLAGS`
- Add missing files or use `foreign` option in `AM_INIT_AUTOMAKE`
- Use portable Make syntax (avoid GNU-specific extensions unless documented)
- Fix conditional usage and variable assignments

### 3. shellcheck

**Purpose**: Static analysis of shell scripts for common errors and anti-patterns

**Command**: `shellcheck -e SC1091,SC2148 <script>`

**Excluded Checks**:
- `SC1091`: Not following sourced files (many scripts source build-common.sh)
- `SC2148`: Tips for adding shebang (handled by project conventions)

**Checks for**:
- Unquoted variables that could cause word splitting
- Use of undefined variables
- Incorrect use of `[` vs `[[`
- Missing error handling
- Potential race conditions
- Incorrect command substitution
- Portability issues

**Common Issues**:
- `SC2086`: Double quote to prevent word splitting - `"$variable"` instead of `$variable`
- `SC2046`: Quote command substitution - `"$(command)"` instead of `$(command)`
- `SC2164`: Use `cd ... || exit` to handle cd failures
- `SC2154`: Variable is referenced but not assigned
- `SC2034`: Variable appears unused

**Remediation**:
- Quote all variable expansions unless word splitting is explicitly intended
- Use `set -e` and `set -u` for safer scripts
- Add error handling for critical operations (cd, cp, mkdir)
- Remove unused variables or mark them with `# shellcheck disable=SC2034`

### 4. autolint (Optional)

**Purpose**: Additional autoconf/automake linting (if available)

**Note**: `autolint` is not installed by default in the build environment. If needed, it can be installed from CPAN or distribution packages.

**Command**: `autolint <directory>`

**Checks for**:
- Additional autoconf macro issues
- Automake best practices
- Build system hygiene

## Drift Checking

### Purpose

Drift checking ensures that autogenerated files (configure, Makefile.in, aclocal.m4) that are tracked in version control remain in sync with their source files (configure.ac, Makefile.am). This prevents build failures caused by manual edits to generated files or forgetting to regenerate files after source changes.

### Implementation

The drift check works by:

1. Backing up current autogenerated files (configure, Makefile.in, aclocal.m4)
2. Running `autoreconf -fi` to regenerate files from sources
3. Comparing regenerated files with the backed-up originals
4. Restoring the original files (to avoid modifying the working tree)
5. Reporting drift if differences are detected

### Command

```bash
check_autotools_drift <library_src_dir>
```

### When Drift Occurs

Drift can occur when:
- Source files (configure.ac, Makefile.am) are modified but `autoreconf` is not run
- Autogenerated files are manually edited (should never be done)
- Different autotools versions generate different output
- Timestamps or version strings in generated files differ

### Remediation

If drift is detected:

1. Verify that you're using the correct autotools version:
   - For binutils/gcc/gdb/newlib: Use autoconf 2.69 exactly
   - For other components: Use automake 1.16+ and current autoconf

2. Regenerate files:
   ```bash
   cd <library_src_dir>
   autoreconf -fi
   ```

3. Review the changes:
   ```bash
   git diff
   ```

4. If changes are expected (e.g., after modifying configure.ac), commit them:
   ```bash
   git add configure Makefile.in aclocal.m4
   git commit -m "Regenerate autotools files after configure.ac changes"
   ```

5. If changes are unexpected, investigate:
   - Check autotools versions
   - Verify no manual edits to generated files
   - Check for environment differences

## Integration into Build Process

### Build Script Integration

The linting and drift checks are integrated into the build process via functions in `build-common.sh`:

- `lint_autotools <dir> [--strict]`: Run all linting tools on a directory
  - Default mode: Informational (reports issues but doesn't fail)
  - Strict mode: Fails on any warnings or errors
- `check_autotools_drift <dir>`: Check for drift in autogenerated files
  - Always fails on drift detection (strict mode)

These functions can be called for each component that uses autotools during the build.

### CI Workflow Integration

The CI workflow (`.github/workflows/build.yml`) includes dedicated linting and drift-check steps that run before the main build:

1. **Lint Shell Scripts**: Check all .sh files with shellcheck (strict mode)
2. **Lint Prerequisites**: Check GMP, MPFR, MPC, ISL, Expat for autotools issues (informational mode)
3. **Drift Check**: Verify tracked autogenerated files are in sync (strict mode)
4. **Build**: Proceeds even if linting finds issues (in informational mode)

Additionally, a separate CodeQL workflow (`.github/workflows/codeql.yml`) provides security-focused analysis:
- Runs on push, pull requests, and weekly schedule
- Performs shellcheck security analysis on all shell scripts
- Uploads results to GitHub Security tab

### Failure Behavior

**Current Implementation (Informational Mode)**:
- Autotools linting reports issues but does NOT fail the build
- Drift detection DOES fail the build with instructions to regenerate files
- Shell script linting (shellcheck) DOES fail the build on errors
- This allows gradual improvement without blocking development

**Future Migration Path**:
As components are modernized and cleaned up, they can be switched to strict mode:
```bash
# In CI workflow
lint_autotools "src/$lib" --strict
```

## Component-Specific Notes

### Prerequisites (GMP, MPFR, MPC, ISL, Expat)

These libraries use standard automake and autoconf:
- **Autogenerated files**: NOT tracked in git (regenerated on each build)
- **Linting**: Full autoconf and automake linting applied
- **Drift checking**: Skipped (files are always regenerated)

### Main Components (binutils, gcc, gdb, newlib)

These components have special requirements:
- **Autoconf version**: Exactly 2.69 required for reproducible builds
- **Autogen**: Use `autogen Makefile.def` to generate Makefile.in before running autoconf
- **Autogenerated files**: Many tracked in git (especially subdirectories)
- **Linting**: Applied with autoconf 2.69
- **Drift checking**: Applied to tracked files

### Shell Scripts

All `.sh` build scripts are checked with shellcheck:
- **build-common.sh**: Core build functions
- **build-prerequisites.sh**: Library building
- **build-binutils-gcc-first.sh**: Binutils and GCC first pass
- **build-newlib.sh**: Newlib C library
- **build-gcc-final-gdb.sh**: GCC final and GDB
- **build-runtime-libs-finalize.sh**: Finalization

## Usage Examples

### Linting a Component

```bash
# Source the build-common.sh functions
source build-common.sh

# Lint a specific component
lint_autotools src/gmp
lint_autotools src/binutils
```

### Checking for Drift

```bash
# Check if autogenerated files are in sync
check_autotools_drift src/newlib
check_autotools_drift src/gdb
```

### Manual Linting

You can also run the tools manually:

```bash
# Autoconf linting
cd src/gmp
autoconf --warnings=all -o /dev/null

# Automake linting
cd src/mpfr
automake --warnings=all --dry-run

# Shellcheck
shellcheck build-prerequisites.sh
```

## Troubleshooting

### Issue: autoconf warnings about deprecated macros

**Solution**: Update configure.ac to use modern equivalents. Consult autoconf documentation for migration guide.

### Issue: automake warnings about INCLUDES

**Solution**: Replace `INCLUDES = ...` with `AM_CPPFLAGS = ...` in Makefile.am

### Issue: Drift detected but no source changes

**Solution**: Check autotools version. For binutils/gcc/gdb/newlib, ensure autoconf 2.69 is being used.

### Issue: Shellcheck warnings about unquoted variables

**Solution**: Quote variables: `"$VARIABLE"` instead of `$VARIABLE`

### Issue: Build fails with "autoconf2.69 not found"

**Solution**: Install autoconf2.69 package (Ubuntu: `apt-get install autoconf2.69`)

## Version Requirements

- **autoconf**: 2.69 for binutils/gcc/gdb/newlib, 2.71+ for other components
- **automake**: 1.16.5 or later
- **shellcheck**: 0.9.0 or later (for shell script analysis)
- **autogen**: Any recent version (for Makefile.def processing)
