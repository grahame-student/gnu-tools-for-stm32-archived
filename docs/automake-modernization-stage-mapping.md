# Automake Modernization Stage Mapping

## Overview

This document provides a comprehensive mapping of the Dockerfile build stages, identifying:
1. The tools and libraries built in each stage
2. How autotools (autoconf/automake/autogen/libtool) are used in each stage
3. Autogenerated files currently tracked in version control

## Build Stage Summary

The toolchain is built using a multi-stage Docker build with the following stages:

```
bootstrap → binutils-gcc-first → newlib → gcc-final-gdb → runtime-libs → main
```

---

## Autotools Version Requirements

### Critical Version Constraints
- **autoconf 2.69** - Required for binutils/gcc/gdb for reproducible builds
  - Ubuntu 22.04+ ships with autoconf 2.71 by default
  - `autoconf2.69` package must be installed explicitly
  - `regenerate_autotools()` detects and uses autoconf2.69 for these components

### Minimum Versions
- **automake 1.16+** - First available in Ubuntu 22.04
- **autogen** - Any version (for Makefile.def processing)
- **libtool** - Any recent version

## Regeneration Strategy

### Currently Regenerated (Not Tracked in Git)
- **gmp** - All autotools files regenerated via `regenerate_autotools()`
- **mpfr** - All autotools files regenerated via `regenerate_autotools()`
- **mpc** - All autotools files regenerated via `regenerate_autotools()`
- **isl** - Mostly regenerated (except isl_config.h.in)
- **expat** - Mostly regenerated (except expat_config.h.in)
- **libiconv** - Mostly regenerated (except Makefile.in files)
- **binutils** - Top-level configure/Makefile.in regenerated (subdirectories tracked)
- **gcc** - Top-level configure/Makefile.in regenerated (subdirectories tracked)

### Currently Tracked in Git (Not Regenerated)
- **gdb** - configure, Makefile.in, and many subdirectory files tracked
- **newlib** - configure, Makefile.in, and subdirectory files tracked
- **zlib** - configure and Makefile.in tracked (custom, not autotools)

### The `regenerate_autotools()` Function
Located in `build-common.sh` (lines 256-388), this function:
1. Checks if configure and Makefile.in already exist (skip if present)
2. For binutils/gcc/gdb: runs `autogen Makefile.def` to generate Makefile.in
3. Ensures autoconf 2.69 is used for binutils/gcc/gdb
4. Runs `autoreconf -i -f` to regenerate all autotools files
5. Installs auxiliary files from automake
6. Recursively regenerates subdirectories with configure.ac
7. Special handling for libiconv (custom ACLOCAL_PATH)

## Recommendations for Modernization

### Phase 1: Continue Current Approach
- Keep prerequisite libraries (gmp, mpfr, mpc, isl, expat) regenerating autotools files
- This is working well and reduces repository size

### Phase 2: Evaluate Main Components
Consider whether to:
1. Continue tracking gdb/newlib autogenerated files (current approach)
2. Regenerate gdb/newlib autogenerated files (like binutils/gcc)

Trade-offs:
- **Tracking files**: Faster builds, guaranteed reproducibility, but larger repo
- **Regenerating files**: Smaller repo, but requires autotools installed, slight risk of variation

### Phase 3: Document Autotools Version Lock
- Document that binutils/gcc/gdb require exactly autoconf 2.69
- Ensure build environment always uses correct version
- Consider adding version checks to build scripts
