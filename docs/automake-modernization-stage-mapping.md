# Automake Modernization Stage Mapping

## Overview

This document provides a comprehensive mapping of the Dockerfile build stages, identifying:
1. The tools and libraries built in each stage
2. How autotools (autoconf/automake/autogen/libtool) are used in each stage
3. Autogenerated files currently tracked in version control

## Build Stage Summary

The toolchain is built using a multi-stage Docker build with the following stages:

```
bootstrap → binutils-gcc-first → newlib → gcc-final-gdb → runtime-libs → main
```

---

5. Installs auxiliary files from automake
6. Recursively regenerates subdirectories with configure.ac
7. Special handling for libiconv (custom ACLOCAL_PATH)

## Recommendations for Modernization

### Phase 1: Continue Current Approach
- Keep prerequisite libraries (gmp, mpfr, mpc, isl, expat) regenerating autotools files
- This is working well and reduces repository size

### Phase 2: Evaluate Main Components
Consider whether to:
1. Continue tracking gdb/newlib autogenerated files (current approach)
2. Regenerate gdb/newlib autogenerated files (like binutils/gcc)

Trade-offs:
- **Tracking files**: Faster builds, guaranteed reproducibility, but larger repo
- **Regenerating files**: Smaller repo, but requires autotools installed, slight risk of variation

### Phase 3: Maintain Autotools Version Compatibility

**Current State:**
- binutils/gcc/gdb require exactly autoconf 2.69 for reproducible builds
- Build environment uses autoconf2.69 package explicitly
- Version detection logic in `regenerate_autotools()` handles version switching

**Upgrade Path:**

When newer autotools versions become available:

1. **Monitor Upstream Requirements**
   - Track binutils/gcc/gdb release notes for autotools version updates
   - Subscribe to autotools mailing lists for deprecation notices
   - Review Ubuntu LTS releases for default version changes

2. **Testing Strategy**
   - Create a test branch with updated autotools versions
   - Regenerate all autogenerated files and compare diffs
   - Run full build in Docker to catch any issues
   - Test on all supported platforms (Linux, macOS if applicable)

3. **Gradual Migration**
   - Update prerequisite libraries first (GMP, MPFR, MPC, ISL, Expat)
   - Test extensively before updating main components
   - Only update binutils/gcc/gdb when upstream confirms compatibility
   - Document any workarounds needed during transition

4. **Version Pinning Strategy**
   - Pin exact autotools versions in Dockerfile for reproducibility
   - Use apt-get package version syntax: `autoconf=2.71-2`
   - Document why specific versions are pinned
   - Review pins annually or when upgrading Ubuntu base image

5. **Rollback Plan**
   - Keep old Dockerfile in git history for reference
   - Document known-good version combinations
   - Maintain ability to build with older autotools versions
   - Consider maintaining branches for different Ubuntu versions

**Example Upgrade Workflow:**

```bash
# 1. Create test environment
docker build --build-arg AUTOCONF_VERSION=2.71 --target bootstrap -t test-upgrade .

# 2. Test regeneration
docker run -it test-upgrade bash
cd /root/build/gnu-tools-for-stm32/src/gmp
rm -rf configure Makefile.in aclocal.m4
autoreconf -i -f
# Check for errors

# 3. Compare outputs
diff <(git show HEAD:configure) configure || echo "Changes detected - review needed"

# 4. Document changes
git add docs/upgrade-notes-autoconf-2.71.md
git commit -m "Document autoconf 2.71 upgrade process"
```
